UI & Server

The two halves of every Shiny app

Shiny app anatomy

  1. UI (user interface)
    • what the user sees and interacts with
  2. Server
    • the R code that runs things

. . .

  • The two communicate through named inputs and outputs
  • shinyApp(ui, server) wires them together and launches the app
  • The app is always named app.R

ui

  • The ui defines the look and feel of the app
  • Defines inputs - widgets the user controls
    • for the server, where functions are actually evaluated
  • Declares where outputs will be rendered
  • Built from nested R functions that generate HTML

ui

  • Every input needs a unique inputId
  • Here we have defined a sliderInput, which we’re calling bins
    • The label is what will be displayed
    • It will take on values from 1 to 50, and will start at 30 by default
  • Access the specific value the user selects within the server through input$inputId
    • input$bins
# Define UI for application that draws a histogram
ui <- fluidPage(

    # Application title
    titlePanel("Old Faithful Geyser Data"),

    # Sidebar with a slider input for number of bins 
    sidebarLayout(
        sidebarPanel(
            sliderInput(
                        inputId = "bins",
                        label = "Number of bins:",
                        min = 1,
                        max = 50,
                        value = 30)
        ),

        # Show a plot of the generated distribution
        mainPanel(
           plotOutput("distPlot")
        )
    )
)

Widgets

All share a common setup:

widgetInput(inputId, label, ...)

  • sliderInput — continuous range
  • radioButtons — discrete choices, all visible at once
  • selectInput — dropdown for many choices
  • checkboxGroupInput — multiple selections

Change the input

  • Let’s say instead of a slider, we want 5 options:
    • 1, 5, 20, 50, or 100
  • We can change the input from a slider (representing continuous intervals)
    to radio buttons, representing discrete choices
  • Just swap the widget — input$bins still works in the server without any other changes!
  • Just change sliderInput to radioButtons, and put in the appropriate choices
    • You can also add a selected argument for the default (starting) value
  • Be sure to read the ?radioButtons help documentation!

    sidebarLayout(
        sidebarPanel(
            sliderInput("bins",
                        "Number of bins:",
                        min = 1,
                        max = 50,
                        value = 30),

            radioButtons("facet_var",
                         "Facet by:",
                         choices  = c("Year" = "year",
                                      "Transmission" = "trans",
                                      "Vehicle Class" = "class"),
                         selected = "year")

        ),

        # Show a plot of the generated distribution
        mainPanel(
           plotOutput("distPlot")
        )

# Define server logic required to draw a histogram
server <- function(input, output) {

    output$distPlot <- renderPlot({

        ggplot2::ggplot(mpg, aes(hwy)) +
          geom_histogram(bins = input$bins, fill = "darkgray") +
          facet_wrap(vars(!!sym(input$facet_var))) +
          labs(
            x = 'Miles per Gallon (MPG)',
            title = 'Distribution of Highway MPG'
          ) +
          theme_minimal()
    })
}

# Run the application
shinyApp(ui = ui, server = server)

But first…NSE 🙄

Working with {dplyr} in Shiny

Can’t use a column name from input$var directly with {dplyr}

. . .

This fails

output$table <- renderReactable({
  mpg |>
    group_by(input$var) |>
    summarise(
      mean = mean(hwy),
      sd   = sd(hwy),
      min  = min(hwy),
      max  = max(hwy)
    )
})

Why does it fail?

  • In non-Shiny {dplyr} code, curly-curly {{}} passes column names from function arguments
  • {{}} is for function arguments
  • But input$var is a "character" not an argument so we need a different approach
  • {dplyr} (with NSE) expects a column name, not a "character"

The Fix

Use !!sym() instead of {{}}

  • sym() turns the class:"character" into the symbol class
  • !! (bang-bang) tells {dplyr} to evaluate it
    • “use the value, not the name”

. . .

This works

output$table <- renderReactable({
  mpg |>
    group_by(!!sym(input$var)) |>
    summarise(
      mean = mean(hwy),
      sd   = sd(hwy),
      min  = min(hwy),
      max  = max(hwy)
    )
})

    • In this case it’s not that big of a deal
    • Rather than curly-curly {{}}

      • sym() transforms a string to a symbol (or name)
      • !! evaluates that symbol (bang-bang)

Layouts

Organizing the app’s interface

Layout Options

  • sidebarLayout() — controls on left, output on right
    • most common for simple apps; wrap in fluidPage()
  • tabsetPanel() — tabs within a panel
    • great for separating a plot tab from a table tab
  • navbarPage() — full navigation bar across the top
    • Each tabPanel() is a separate page with its own sidebar
    • Don’t share inputId names across pages in navbarPage()

The Bootstrap Grid

Shiny uses a 12-column grid system under the hood

Every row has 12 units to divide up how you want

  • fluidRow() + column()
    • fluidRow(column())
    • fine-grained control
    • column(width = 4, ...) = 1/3 of the row
fluidRow(
  column(width = 4, plotOutput("plot1")),   
  column(width = 4, plotOutput("plot2")),   
  column(width = 4, plotOutput("plot3"))    
)

The Bootstrap Grid

Remember

  • the number is a share of 12, not a pixel width or a percentage
  • As long as the widths in a fluidRow() add up to 12, the columns fill the row evenly
  • If they add up to less than 12, the remaining space is just left empty
    • which is sometimes useful for centering things
Layout Code
Half / Half column(6) + column(6)
Sidebar / Main column(3) + column(9)
Thirds column(4) + column(4) + column(4)
Quarters column(3) + column(3) + column(3) + column(3)

Reactivity

How Shiny knows what to update

Simple example

Try this app. Type the letters in one at a time. Notice how it updates.

library(shiny)

ui <- fluidPage(
  textInput("name", "What's your name?"),
  textOutput("greeting")

server <- function(input, output, session) {
  output$greeting <- renderText({

Mastering Shiny

Programming style

  • Notice you don’t have to “run” the code each time the input updates
  • Your app provides instructions to R and Shiny decides when it actually runs the code

. . .

This is known as declarative programming

. . .

Declarative programming means you provide instructions, but don’t actually run it

. . .

Normal R code is imperative programming - you decide when it’s run

What is reactivity?

  • Normal R is imperative
    • you control when code runs
  • Shiny uses declarative programming
    • you describe relationships, not sequences
    • you describe a dependency graph and Shiny decides when to re-run
    • when input$bins changes → anything that depends on it is automatically re-executed
  • Think of it like a spreadsheet: change a cell, formulas update automatically

Key reactive functions

  • render*() — creates a reactive output
  • reactive() — creates a reusable reactive value/expression
  • observe() — runs side effects (e.g., writing to a file)
  • observeEvent() — runs when a specific input changes

Reactivity

  • Shiny builds a graph of dependencies at runtime
  • input → reactive expression → output
  • When an input changes, only the affected outputs re-execute
  • You can visualize this with the {reactlog} package

Inputs

input is a basically a list object that contains objects from the ui

. . .

ui <- fluidPage(
  numericInput("count", label = "Number of values", value = 100)
)

After writing this code, input$count will be a available in the server, and the value it takes will depend on the browser input (starting at 100)

These are read-only, and cannot be modified

Selective read permissions

It must be in a reactive context, or it won’t work

That’s why this results in an error - trying to print() in the server won’t work outside a reactive context

server <- function(input, output, session) {
  print(paste0("The value of input$count is ", input$count)) 
}

shinyApp(ui, server)
# > Error in .getReactiveEnvironment()$currentContext() : 
# >  Operation not allowed without an active reactive context. 
# > (You tried to do something that can only be done from inside a reactive expression or observer.)

Output

  • The output object is similar to input, in terms of being a list-like object
  • Create new components of the list for new output, and refer to them in the ui
  • These also need to be in reactive contexts (e.g., render*())

reactive() tips

  • If two outputs need the same data, don’t repeat yourself
  • Wrap shared computation in reactive() — it’s cached and only re-runs when inputs change
  • Call it like a function (with parentheses)
    • filtered_data()
  • But never call reactive expressions outside a reactive context
server <- function(input, output, session) {

  # Shared reactive expression — runs once, used by both outputs
  filtered_data <- reactive({
    mpg |> filter(class == !!sym(input$class))
  })

  output$plot  <- renderPlot({
    ggplot(filtered_data(), aes(displ, hwy)) + geom_point()
  })

  output$table <- renderTable({
    filtered_data()
  })
}

Why reactivity?

Imagine we want to have a simple app converting temperatures from Fahrenheit to Celsius

. . .

Do this with variables

temp_f <- 72
temp_c <- (temp_f - 32) * (5/9)
temp_c
[1] 22.22222

. . .

But changing temp_f has no impact on temp_c

temp_f <- 50
temp_c
[1] 22.22222

Use a function?

Let’s instead make this a function that depends on the object in the global environment.

to_celsius <- function() {
  (temp_f - 32) * (5/9)
}
to_celsius()
[1] 10
temp_f <- 30
to_celsius()
[1] -1.111111

. . .

This works, but it’s less than ideal computationally.

Even if temp_f hasn’t changed, the conversion is re-computed

Not a big deal in this case, but often is.

Reactive alternative

First create a reactive variable

library(shiny)
reactiveConsole(TRUE)
temp_f <- reactiveVal(72)
temp_f()
[1] 72
temp_f(50)
temp_f()
[1] 50

Reactive function

Next create a reactive function

to_celsius <- reactive({
  message("Converting...")
  (temp_f() - 32) * (5/9)
})

. . .

Now it will convert only when the value of temp_f() changes

Reactive function

Unless the value of temp_f changes, the code will not be re-run.

to_celsius()
Converting...
[1] 10

. . .

to_celsius()
[1] 10

. . .

temp_f(100)
to_celsius()
Converting...
[1] 37.77778

. . .

to_celsius()
[1] 37.77778

. . .

temp_f(75)
to_celsius()
Converting...
[1] 23.88889

  • This doesn’t work with shiny
  • Instead, we think through reactive graphs

Tutorial

Deploy to Connect Cloud

  1. Make sure your RStudio Project is connected to Github
  2. Build your Shiny app
  3. Preview the app
    • In RStudio when looking at app.R, click the Run button
    • This previews your application locally
  4. Add dependency file
    • Create a dependency file so Connect Cloud can rebuild your content during deployment
    • Load the {rsconnect} package and run writeManifest()

rsconnect::writeManifest()

Deploy to Connect Cloud

After you writeManifest()

  1. Push to Github

Deploy to Connect Cloud

  1. Deploy to Posit Connect Cloud
    1. Sign in to Connect Cloud
    2. Click the “Publish” icon button on the top of your Home page
    3. Select Shiny
    4. Select the public repository that you created
    5. Confirm the branch
    6. Select app.R as the primary file
    7. Click “Publish”
      • will display status updates during the deployment process
      • also build logs will stream on the lower part of the screen
    8. 🎉 Share the link!
